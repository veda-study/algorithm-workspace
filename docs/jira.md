# 🛠️ Jira 활용 및 데일리 로그 가이드 (Jira Guide)

> VEDA 코딩 테스트 스터디는 한화 비전에서 가장 활발히 사용하는 협업 툴인 **Jira**를 기반으로 운영됩니다.
> 단순히 일정을 기록하는 것을 넘어, 매일의 고민과 접근 방식을 기록하여 현업 수준의 애자일(Agile) 프로세스를 경험하는 것을 목표로 합니다.

## 📌 1. Jira 이슈 구조 및 내 카드 찾기

우리는 같은 문제를 각자 풀기 때문에, 번호가 겹치거나 진행 상황이 꼬이는 것을 방지하기 위해 **하위 작업(Sub-task)** 체제로 운영합니다.

1. **상위 작업 (Task):** 스터디 리더가 주차별로 풀 문제를 생성합니다. (예: `[Week 01] 프로그래머스 PCCE 기출문제 1번 문자 출력` ➡️ 이슈 키: `VEDA-1`)
2. **하위 작업 (Sub-task):** 상위 작업 안에 멤버별 개인 카드가 생성됩니다. (예: `명준 - 프로그래머스 PCCE 기출문제 1번 풀이` ➡️ 이슈 키: `VEDA-7`)
3. **주의 사항:** `git commit`을 하거나 데일리 로그를 작성할 때는 **반드시 상위 에픽이 아닌 '본인 이름이 적힌 하위 스토리의 이슈 키'**(예: VEDA-7)를 사용해야 합니다.

> 💡 **[필독] Jira 이슈 번호(Key)에 대한 안내**
> * Jira는 시스템 구조상 `VEDA-1-1`, `VEDA-1-2` 같은 하위 번호를 지원하지 않고, 어떤 카드든 생성되는 순서대로 프로젝트 전체에서 고유한 번호가 1씩 늘어나며 부여됩니다.
> * 따라서 리더가 문제를 먼저 여러 개 생성해 두면, 상위 문제는 `VEDA-1`인데 내 하위 카드는 `VEDA-7`이 되는 등 **번호가 연속되지 않는 것이 정상**입니다.
> * **핵심 규칙:** 번호가 왜 뛰었는지 신경 쓰지 마세요! 여러분은 오직 jira 칸반 보드에서 **[본인 이름]이 박혀있는 카드 좌측 상단의 이슈 키**만 확인하고, 그 번호로만 커밋과 데일리 로그를 남기시면 됩니다.

## 📊 2. 칸반 보드(Kanban Board) 상태 관리 규칙

매일 공부를 시작하고 마칠 때, 본인 카드의 상태(Column)를 아래 규칙에 맞게 드래그하여 이동시킵니다.

#### 🔄 상태(Status) 관리
* **TO DO (할 일):** 이번 주에 나에게 할당된 풀지 않은 문제들.
* **IN PROGRESS (진행 중):** 내가 현재 오늘 치열하게 고민하고 코드를 짜고 있는 문제.
* **IN REVIEW (리뷰 대기 🔍):** 문제를 최종 완료하고 GitHub에 PR(Pull Request)을 올려 팀원들의 코드 리뷰를 기다리는 상태.
* **DONE (완료 🎉):** 팀원의 Approve를 받고 `main` 브랜치에 Squash Merge까지 최종 완료된 상태.

#### 🏷️ 라벨(Label)을 통한 상태 공유 (SOS 요청)
현업에서 막힌 부분을 투명하게 공유하는 것은 필수입니다. 상황에 맞는 라벨을 달아 현재 상태를 팀원들에게 알리세요. (스터디 공통 라벨은 지라에 이미 생성되어 있습니다!)

* **🆘헬프요청**: 2시간 이상 고민했는데 답이 안 나올 때 (리뷰어 및 리더가 우선적으로 확인하여 힌트 제공)
* **⏱️시간초과**: 로직은 맞는데 효율성 테스트를 통과하지 못할 때
* **🔄복습필요**: 남의 코드를 참고해서 풀었거나, 나중에 꼭 다시 풀어봐야 할 때

**📌 라벨 적용하는 방법**
1. 칸반 보드에서 내 카드(하위 작업)를 클릭하여 엽니다.
2. 우측 세부 정보(Details) 사이드바에서 **`레이블 추가`** 항목을 찾습니다.
3. 빈칸을 클릭한 후, 위 약속된 라벨 이름(예: `🆘헬프요청`)을 입력하고 엔터를 누르거나 목록에서 선택합니다.

## 📝 3. [필수] 데일리 로그(일일 요약본) 작성 규칙

> **"구현을 완료하지 못했더라도, 오늘 어디까지 접근했고 어디서 막혔는지 기록하는 것이 개발자의 핵심 역량입니다."**

문제를 당일에 마스터하지 못했더라도, **매일 공부를 마칠 때(24:00 전까지) 본인의 Jira 하위 작업 카드 내부 `댓글(Comment)` 공간**에 아래 양식으로 데일리 로그를 반드시 작성합니다.

### 📋 데일리 로그 작성 템플릿 (복사해서 댓글에 사용)
```markdown
### 📅 [2026.05.18] 데일리 진행 상황
* **현재 상태:** ⏳ 진행 중 (또는 ⚠️ 트러블슈팅 중 / 🎉 완료)

* **오늘의 핵심 접근 방식:**
  * (예시: 괄호 회전 문제를 풀기 위해 처음에는 단순 배열 순회로 접근함.)

* **🔥 트러블슈팅 및 알게 된 점 (Problem & Solution):**
  * **문제 발생:** 
    - 배열의 요소를 매번 앞으로 당기다 보니 문자열이 길어질수록 시간 초과(TLE) 발생.

  * **접근 방안 1: Stack을 활용한 검증**
    - **근거:** 괄호의 짝을 맞추는 전형적인 문제이므로, LIFO(후입선출) 구조인 Stack이 적합할 것이라 판단함.
    - **결과:** ❌ 실패 (여전히 회전 시 O(N^2)의 시간이 걸려 일부 케이스에서 시간 초과)

  * **접근 방안 2: Queue를 활용한 회전 + Stack 검증 조합**
    - **근거:** 요소를 회전시키는 작업은 맨 앞의 데이터를 뽑아 맨 뒤로 넣는 FIFO 구조이므로, Queue의 `pop()`과 `push()`를 쓰면 회전 비용을 O(1)로 줄일 수 있음.
    - **결과:** ✅ 성공 (기존 접근 대비 실행 속도 약 0.5초 단축, 메모리 사용량도 제한 범위 내 통과)

* **💡 내일의 목표 (또는 느낀 점):**
  * 자료구조를 조합(Queue + Stack)해서 사용하면 시간 복잡도를 크게 줄일 수 있다는 것을 체감함. 내일은 코드를 더 간결하게 리팩토링할 예정.
```

## 🔗 4. GitHub 자동 연동 확인하기

우리가 세팅한 GitHub 연동 파이프라인 덕분에, 여러분이 git-flow 규칙에 맞춰 커밋하고 PR을 올리면 Jira가 알아서 반응합니다.

#### ⚙️ 기본 자동화 연동
* **브랜치/PR 연동**: GitHub에서 PR을 올리면 Jira 카드가 자동으로 IN PROGRESS ➡️ IN REVIEW로 이동합니다. Merge 시 DONE으로 이동합니다.

* **커밋 연동**: 커밋 메시지에 [VEDA-7]를 넣으면, Jira 카드 우측 개발(Development) 세션에 커밋 내역이 바로 꽂힙니다.

#### ⏱️ 스마트 커밋 (Smart Commits)으로 시간 추적하기

Jira에 직접 접속하지 않아도, 커밋 메시지 하나로 소요 시간(Time Tracking)을 기록하고 댓글을 달 수 있습니다. 문제당 소요 시간이 어떻게 단축되고 있는지 기록해 보세요.

* 사용법: 커밋 메시지 뒤에 #time과 #comment 명령어를 추가합니다.
* 예시:
```bash
git commit -m "feat: [VEDA-7] 문제 1번 풀이 완료 #time 2h 30m #comment Stack을 썼더니 효율성 통과함!"
```
